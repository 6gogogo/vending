# 完整本地联调流程

这份说明专门回答一个问题：

“我怎样在本地虚拟一个柜机，用小程序领取物资，同时在电脑后台看到柜机和事件状态变化？”

## 1. 先把三个部分分清楚

你当前项目里有 3 个主要成果：

- `apps/api`
  这是你的业务后端。
  它负责登录、规则校验、开柜事件落库、库存记录、回调处理、后台统计。
  它不是写进柜机 Android 系统里的程序。

- `apps/mobile`
  这是给用户和商户使用的移动端。
  同一套代码会编译成微信小程序、支付宝小程序和 App。

- `apps/admin-web`
  这是电脑后台页面。
  它本身只是前端页面，真正的数据仍然来自 `apps/api`。

柜机厂商接口在当前仓库里对应的是：

- `apps/api/src/modules/devices/smartvm.gateway.ts`
- `sandbox/`

也就是说：

- `apps/api` 是你的业务后端
- `sandbox` 用来模拟“柜机回调”和“柜机接口调试”
- 真正的柜机 Android 程序本轮没有实现成独立工程

## 2. 当前本地联调最推荐的模式

当前仓库最稳的本地闭环是：

1. 启动 `apps/api`
2. 启动 `apps/admin-web`
3. 用 `apps/mobile` 编译微信小程序
4. 在小程序里点“开柜”
5. 用 `sandbox` 模拟柜机回调
6. 在后台“柜机监控”页看设备、事件、回调日志变化

## 3. 启动顺序

### 可选：完全清空后端数据

如果你要从真正空库开始，而不是恢复到项目最初的调试种子数据，先停止后端，再执行：

```powershell
npm run init:all-data
```

这会清空业务数据、系统审计日志和上传目录，并且不会在后端重启时自动补回测试柜机。执行后再继续下面的启动步骤。

### 第一步：启动后端

```powershell
$env:PORT=4000
npm run dev:api:once
```

如果 `4000` 被占用，就换端口，例如：

```powershell
$env:PORT=4001
npm run dev:api:once
```

## 4. 启动电脑后台

新开一个终端：

```powershell
npm run dev:admin
```

然后打开后台页面，进入：

```text
/operations
```

这个页面会每 3 秒自动刷新一次，用来监控：

- 柜机状态
- 商品库存
- 开柜事件
- 回调日志

## 5. 配置移动端接口地址

### 开发者工具模拟器

如果你只在电脑里的微信开发者工具模拟器看页面，可以用：

```text
http://127.0.0.1:4000/api
```

### 微信真机调试

如果你用手机真机调试，**绝对不能继续写 `localhost` 或 `127.0.0.1`**。

因为这时小程序运行在手机上：

- `localhost` 指向的是手机自己
- 不是你的电脑

所以你必须把接口地址改成你电脑的局域网 IP，例如：

```text
http://192.168.1.20:4000/api
```

你可以在 `apps/mobile` 下新建：

```text
.env.local
```

内容例如：

```env
VITE_API_BASE_URL=http://192.168.1.20:4000/api
```

`apps/mobile/.env.example` 里已经给了格式示例。

## 6. 构建并导入微信小程序

```powershell
npm run build:mobile:weixin
```

然后导入：

```text
apps/mobile/dist/build/mp-weixin
```

登录测试账号：

- 特殊群体：`13800000002`
- 商户：`13800000004`
- 验证码：先点“获取验证码”，当前预览码固定是 `123456`

## 7. 虚拟建立一个柜机

当前你有两种方式。

### 方式 A：直接用内置种子柜机

项目默认已经有两个柜机：

- `CAB-1001`
- `CAB-1002`

它们定义在：

- `packages/shared-types/src/index.ts`

### 方式 B：用 sandbox 创建或覆盖一个模拟柜机

先编辑：

```text
sandbox/fixtures/mock-device.sample.json
```

你可以把里面的商品和库存改成你需要的数量。

例如：

- 柜机编号
- 柜机名称
- 商品名称
- 商品分类
- `stock`

然后执行：

```powershell
npm run sandbox:mock-device
```

这个命令会调用本地后端，把模拟柜机写进当前内存数据里。

注意：

- 当前是内存仓储
- 你一旦重启后端，模拟柜机会恢复成种子状态

## 8. 用小程序发起开柜

### 特殊群体领取流程

1. 用 `13800000002` 登录
2. 进入“公益领取”首页
3. 点某个柜机卡片里的“申请开柜领取”

点击后，移动端会调用：

- `POST /api/cabinet-events/open`

后端会先创建一条开柜事件。

当前页面也会显示：

- 最近开柜单号
- 最近事件编号

### 商户投放流程

1. 用 `13800000004` 登录
2. 进入“商户中心”
3. 点“开柜”

商户走完回调后，库存会增加，并可能产生过期预警。

## 9. 模拟柜机回调

### 最简单的方法

当你已经在小程序里点完“开柜”后，直接执行：

```powershell
npm run sandbox:latest-event
```

这个命令会自动：

1. 读取本地后端里最新的一条开柜事件
2. 自动发送门状态成功回调
3. 自动发送结算回调

适合你做“点击小程序按钮 -> 后台变化”的快速联调。

### 更细粒度的方法

如果你要分别控制门状态和结算，也可以手动执行：

```powershell
npm run sandbox:door-status
npm run sandbox:settlement
```

这两个命令使用固定示例 JSON，适合调单个回调接口。

## 10. 后台应该看到什么

当流程正确时，你应该能在“柜机监控”页看到：

### 设备区

- 柜机在线状态
- 商品库存数量变化

### 事件区

- 新的 `orderNo`
- 新的 `eventId`
- 状态从 `created` 进入 `opened` / `settled`

### 回调区

- `door-status`
- `settlement`

如果是特殊群体领取：

- 库存减少
- 领取记录增加

如果是商户投放：

- 库存增加
- 可能生成预警任务

## 11. 一条最完整的推荐演练链路

你可以严格按这个顺序做：

1. `npm run dev:api:once`
2. `npm run dev:admin`
3. 打开后台 `/operations`
4. `npm run sandbox:mock-device`
5. `npm run build:mobile:weixin`
6. 导入微信开发者工具
7. 用 `13800000002` 登录
8. 点“申请开柜领取”
9. 看到页面显示“最近开柜单号”和“最近事件编号”
10. 回到终端执行 `npm run sandbox:latest-event`
11. 看后台 `/operations` 页面里的事件状态、库存和回调日志变化

## 12. 如果你看到微信开发者工具正常，但真机不正常

优先检查这几项：

1. 小程序构建时的 `VITE_API_BASE_URL` 是不是仍然写着 `localhost`
2. 手机和电脑是不是在同一局域网
3. 后端端口是不是开放
4. 微信开发者工具是否启用了“不校验合法域名、web-view（业务域名）、TLS 版本以及 HTTPS 证书”

当前你这次遇到的 `fetch is not defined` 已经在代码层修复，移动端请求底层已经改成 `uni.request`，不再依赖浏览器原生 `fetch`。

你这次又遇到的 `URL is not defined` 也已经在代码层修复，原因是旧版共享请求层还在直接使用浏览器 `URL` 对象；现在已经改成普通字符串拼接，不再依赖小程序里缺失的运行时对象。

## 13. 如果本地流程还不稳定，怎样先借助测试平台

如果你的目标是先验证“柜机接口本身能不能通”，可以先跳过小程序页面，直接用 `sandbox`。

推荐顺序：

1. 在后端或终端环境里配置 `SMARTVM_BASE_URL=http://pre.smartvm.cn`
2. 配置你自己的 `SMARTVM_CLIENT_ID` 和 `SMARTVM_KEY`
3. 执行 `npm run sandbox:goods`
4. 执行 `npm run sandbox:door`
5. 如需补测支付回写，再执行 `npm run sandbox:payment-success`
6. 如需补测退款，再执行 `npm run sandbox:refund`

这里仍然不建议让小程序直接请求测试平台，因为签名凭据应该只放在后端或本地脚本里，不应该进前端代码。
