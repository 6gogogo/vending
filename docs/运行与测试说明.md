# 运行与测试说明

## 0. 统一配置和测试数据入口

现在统一约定：

- 第三方接口地址和密钥只填一处：`apps/api/.env`
- 后端业务数据只存一处：`apps/api/runtime-data/store.json`
- 重置测试数据只用一条命令：`npm run init:api-data`

### 先复制后端配置模板

```powershell
Copy-Item apps/api/.env.example apps/api/.env
```

你后续主要修改的就是：

```env
PORT=4000
API_DATA_FILE=runtime-data/store.json
UPLOAD_DIR=runtime-uploads
SYSTEM_LOG_FILE=runtime-data/system-audit.ndjson
BUSINESS_TIMEZONE_OFFSET_HOURS=8
BUSINESS_DAY_START_HOUR=4
AMAP_WEB_KEY=
SMARTVM_BASE_URL=http://pre.smartvm.cn
SMARTVM_CLIENT_ID=
SMARTVM_KEY=
SMARTVM_PAYMENT_SUCCESS_PATH=/api/pay/container/paymentSuccess
ALIYUN_SMS_ACCESS_KEY_ID=
ALIYUN_SMS_ACCESS_KEY_SECRET=
ALIYUN_SMS_REGION_ID=cn-hangzhou
ALIYUN_SMS_ENDPOINT=dysmsapi.aliyuncs.com
VERIFICATION_CODE_PROVIDER=mock
VERIFICATION_CODE_PREVIEW_ENABLED=true
```

说明：

- `SMARTVM_*`：柜机平台接口地址和签名凭据
- `AMAP_WEB_KEY`：后台地图选点用的高德 Web Key
- `ALIYUN_SMS_*`：阿里云短信验证码服务的访问密钥与接入区域
- `VERIFICATION_CODE_PROVIDER`：验证码服务模式，`mock` 表示本地测试码，`aliyun` 表示真实短信验证码
- `VERIFICATION_CODE_PREVIEW_ENABLED`：是否在接口返回中附带测试验证码，真实短信模式建议改为 `false`
- `API_DATA_FILE`：后端持久化数据文件位置
- `UPLOAD_DIR`：后台和 PC 端上传图片的实际落盘目录
- `SYSTEM_LOG_FILE`：系统级接口审计日志文件位置，默认记录完整请求流

### AI 助手配置

如果要启用后台里的 `AI 工作台`，默认只需要在：

```text
apps/api/.env
```

里补一项：

```env
OPENAI_API_KEY=你的 OpenAI Key
```

当前后端已经按 OpenAI `chat/completions` 格式封装好了五类能力：

- 单次开柜异常诊断
- AI 运维晨报 / 日报
- AI 补货与布局建议
- AI 反馈分流与回复草稿
- AI 策略优化建议

默认值如下，一般不用改：

```env
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_MODEL=gpt-4.1-mini
OPENAI_TIMEOUT_MS=30000
```

如果你后续改用兼容 OpenAI 协议的中转网关或私有模型服务，再调整 `OPENAI_BASE_URL` 和 `OPENAI_MODEL` 即可，前端页面和调用方式不用改。

### 路径解析规则

现在后端的这两个路径：

- `API_DATA_FILE`
- `UPLOAD_DIR`

默认都按 `apps/api` 目录为基准解析，不再依赖你从哪个目录启动进程。

例如：

```env
API_DATA_FILE=runtime-data/store.json
UPLOAD_DIR=runtime-uploads
```

实际会分别落到：

```text
apps/api/runtime-data/store.json
apps/api/runtime-uploads
```

如果你未来用 PM2、NSSM、systemd 或计划任务启动，并且不想依赖项目目录写入，也可以直接改成绝对路径，例如：

```env
API_DATA_FILE=/data/vending-machine/store.json
UPLOAD_DIR=/data/vending-machine/uploads
```

### 初始化测试数据

如果你想把后端数据重置回默认测试状态，直接运行：

```bash
npm run init:api-data
```

这条命令会用项目种子数据覆盖当前持久化文件，适合联调前回到干净状态。

## 0.1 迁移到新机器前的环境要求

如果后端和 PC 端要迁移到另一台机器，最低要求建议按下面准备：

- Node.js：建议 `22 LTS` 或 `24`
- npm：建议 `10+`
- 可写目录权限：
  - `API_DATA_FILE` 指向的目录必须可写
  - `UPLOAD_DIR` 指向的目录必须可写
- 网络连通：
  - PC 前端浏览器必须能访问后端 `http://<你的地址>:<端口>/api`
  - 如果要用地图，高德 key 必须有效
  - 如果要接真实柜机平台，`SMARTVM_*` 必须配置完整
- 浏览器环境：
  - PC 端后台需要现代浏览器，建议 Chrome / Edge 最新稳定版

如果“没有配过环境”，最少要准备这些变量：

```env
PORT=4000
API_DATA_FILE=runtime-data/store.json
UPLOAD_DIR=runtime-uploads
AMAP_WEB_KEY=
SMARTVM_BASE_URL=
SMARTVM_CLIENT_ID=
SMARTVM_KEY=
SMARTVM_PAYMENT_SUCCESS_PATH=/api/pay/container/paymentSuccess
ALIYUN_SMS_ACCESS_KEY_ID=
ALIYUN_SMS_ACCESS_KEY_SECRET=
ALIYUN_SMS_REGION_ID=cn-hangzhou
ALIYUN_SMS_ENDPOINT=dysmsapi.aliyuncs.com
VERIFICATION_CODE_PROVIDER=mock
VERIFICATION_CODE_PREVIEW_ENABLED=true
```

其中：

- 只做本地业务联调时，`AMAP_WEB_KEY`、`SMARTVM_*` 和 `ALIYUN_SMS_*` 可以先空着
- 要正式用地图选点时，必须填 `AMAP_WEB_KEY`
- 要正式接柜机平台时，必须填完整 `SMARTVM_*`
- 要在 `sandbox` 里测试真实短信验证码时，必须填完整 `ALIYUN_SMS_*`
- 要在后端登录、注册流程里启用真实短信验证码时，把 `VERIFICATION_CODE_PROVIDER` 改成 `aliyun`
- 若改成 `aliyun`，建议把 `VERIFICATION_CODE_PREVIEW_ENABLED` 设为 `false`

## 1. 后端为什么会报缺少 `@nestjs/platform-express`

NestJS 的 `NestFactory.create()` 默认需要一个 HTTP 平台驱动。当前项目使用的是标准 HTTP 服务，所以必须安装 `@nestjs/platform-express`。

## 2. 后端怎么运行

### 常用命令

- 热更新开发：`npm run dev:api`
- 单次启动，不开 watch：`npm run dev:api:once`

### 什么时候用哪个

- 平时开发优先用 `npm run dev:api`
- 如果你的本机文件监听、终端权限或杀毒软件影响了 watch，再退回 `npm run dev:api:once`

### 如果现在报 `EADDRINUSE`

这不再是缺包问题，而是 `4000` 端口已经被别的进程占用了。

当前后端默认端口来自：

```ts
const port = Number(process.env.PORT ?? 4000);
```

排查方式：

```powershell
Get-NetTCPConnection -LocalPort 4000 | Select-Object LocalAddress,LocalPort,OwningProcess,State
```

查看进程：

```powershell
Get-Process -Id <进程ID>
```

如果你不想关掉已有进程，也可以直接换端口启动：

```powershell
$env:PORT=4001
npm run dev:api:once
```

或者：

```powershell
$env:PORT=4001
npm run dev:api
```

- 构建后台：`npm run build --workspace @vm/admin-web`

## 4. 移动端现在到底是什么运行方式

当前 `npm run dev:mobile` 对应的是：

```bash
uni -p h5
```

这表示它运行的是 **H5 版本**，本质上是浏览器网页，不是“模拟一个带手机边框的真机界面”。

所以你现在看到“只有一个背景”的情况，通常意味着：

- 你打开的是 H5 页面
- 页面逻辑可能没加载成功，或者
- 你没有进入正确页面流程

它不是手机模拟器，只是把 uni-app 输出成网页来预览。

### 移动端接口地址该怎么配

移动端默认会读取：

```text
VITE_API_BASE_URL
```

示例文件已经放在：

```text
apps/mobile/.env.example
```

如果你只是用电脑里的微信开发者工具模拟器，并且已经勾选了“不校验合法域名、web-view（业务域名）、TLS 版本以及 HTTPS 证书”，可以写：

```env
VITE_API_BASE_URL=http://127.0.0.1:4000/api
```

如果你还在开发者工具里做局域网联调，也可以改成你电脑的局域网 IP，例如：

```env
VITE_API_BASE_URL=http://192.168.1.20:4000/api
```

但这条不适用于微信小程序真机、体验版和正式版。`fail url not in domain list` 的根因通常是下面几种之一：

- 请求地址还是 `http://...`
- 请求地址是公网 `IP:端口`
- 没有把对应域名配置到微信公众平台“服务器域名”
- 开发者工具没有勾选“不校验合法域名...”

原因是：

- 开发者工具模拟器运行在你电脑上
- 真机运行在手机上
- 手机里的 `localhost` 指向手机自己，不指向你的电脑

微信小程序正式可用的口径应当是：

```env
VITE_API_BASE_URL=https://your-domain.com/api
```

并且在微信公众平台配置：

- 开发管理
- 开发设置
- 服务器域名
- request 合法域名：`https://your-domain.com`

### 当前 App / 小程序入口联调流程

新的移动端入口已经改成：

- 初始页：只显示 `登录 / 注册 / 反馈`
- 登录页：只输入 `手机号 + 验证码`
- 注册页：先选身份，再填写资料
- 审核状态页：只显示 `待审核 / 已驳回`

本地联调顺序建议固定为：

1. 启动后端：`npm run dev:api:once`
2. 确认后端监听 `4000`
3. 在 `apps/mobile/.env.local` 或 `apps/mobile/.env` 中写入：

```env
VITE_API_BASE_URL=http://127.0.0.1:4000/api
```

4. 重新执行小程序或 H5 构建
5. 进入移动端初始页测试：
   - `注册`
   - `审核`
   - `登录`
   - `退出登录`

### 后续切换线上服务器 IP 的方法

后续如果后端换成开放的线上服务器，只改环境变量，不改业务代码。

例如把：

```env
VITE_API_BASE_URL=http://127.0.0.1:4000/api
```

改成：

```env
VITE_API_BASE_URL=https://your-domain.com/api
```

改完后需要：

1. 停掉当前移动端 dev/build 进程
2. 重新执行 `npm run dev:mobile` / `npm run build:mobile:weixin`
3. 重新导入或重新编译小程序产物

### 登录 / 注册 / 审核状态怎么验证

- 未登记手机号：
  先在登录页请求验证码，再点登录，应提示“未登记，请先注册或反馈”
- 待审核手机号：
  先在注册页提交资料，再回登录页登录，应提示“审核中”
- 已驳回手机号：
  在 PC 人员管理中驳回后，再回登录页登录，应提示驳回原因，并可跳回注册页覆盖更新
- 已通过手机号：
  在 PC 人员管理中审核通过后，可重新登录并进入底部四栏主界面

2. 执行 `npm run build:mobile:weixin`

### 真机调试额外检查项

如果开发者工具里正常，但真机白屏或请求失败，优先检查：

1. `VITE_API_BASE_URL` 是不是还写着 `localhost`
2. 手机和电脑是否在同一局域网
3. 后端是否真的在对应端口启动
4. 微信开发者工具是否开启了“不校验合法域名、web-view（业务域名）、TLS 版本以及 HTTPS 证书”

### 如果点击按钮时报 `URL is not defined`

这不是你点错了，而是之前请求层里残留了浏览器环境专用的 `URL` 构造器。

现在已经改成：

- 共享请求客户端手工拼接查询字符串
- 小程序请求层只使用 `uni.request`

如果你还看到旧报错，通常说明微信开发者工具里导入的仍是旧产物。重新执行一次：

```bash
npm run build:mobile:weixin
```

然后重新导入或重新编译 `apps/mobile/dist/build/mp-weixin`。

## 8. App 怎么拿到安装包

### 先生成 App 构建产物

在项目根目录运行：

```bash
npm run build:mobile:app
```

或者：

```bash
npm run build:app --workspace @vm/mobile
```

生成目录是：

```text
apps/mobile/dist/build/app
```

### 这个目录是什么

它是 uni-app 的 App 构建产物目录，包含：

- `manifest.json`
- `app-config.js`
- `app-service.js`
- 页面样式和运行资源

它 **不是直接可安装的 `.apk` 或 `.ipa` 文件**。

### 真正的安装包怎么来

如果你要 Android 安装包，常见做法是：

1. 用 HBuilderX 打开 uni-app 项目
2. 选择“发行”
3. 选择原生 App 云打包
4. 配置 Android 包名、证书、图标、启动图
5. 让 HBuilderX 云打包生成 `.apk` 或 `.aab`

如果你只是先看 App 运行效果，可以先用 HBuilderX 导入：

```text
apps/mobile/dist/build/app
```

如果你要继续做正式发包，建议回到项目源目录 `apps/mobile` 配置完整的 `manifest.json`、证书和图标资源后再打包。

1. 把后端环境变量 `SMARTVM_BASE_URL` 指到 `http://pre.smartvm.cn`

- 小程序还要额外处理请求域名白名单
- 后续正式环境也应该由业务后端统一中转柜机接口

## 11. 高德地图与柜机距离怎么配置

### 后台地图选点

后台地图选点使用高德 Web JS API。先在：

```text
apps/api/.env
```

里配置：

```env
AMAP_WEB_KEY=你的高德WebKey
```

然后启动后台：

```bash
npm run dev:admin
```

进入：

1. 柜机监控
2. 点进某一台柜机详情
3. 点击“设置位置”
4. 在地图中搜索地点或直接点图选点
5. 保存后，柜机的经纬度会写入后端数据

### APP 距离排序

移动端不直接加载高德地图脚本，而是：

1. 小程序或 App 获取当前定位
2. 把当前经纬度传给本地后端
3. 后端基于柜机已保存坐标返回距离
4. APP 在“附近柜机”里按距离从近到远显示

如果用户没开定位权限，页面会退回普通顺序，并显示“未开启定位”。

### 小程序和 App 需要的定位权限

如果你要让移动端稳定显示“我的位置”、按距离排序并查找最近柜机，至少需要这两层权限：

- 小程序 / App 业务权限：允许调用定位能力
- 手机系统权限：允许当前应用访问位置信息

当前项目已经在页面逻辑里调用 `uni.getLocation`。你后续上线或真机测试时，还需要确认：

- 微信小程序已申请并说明 `scope.userLocation` 用途
- Android / iOS 真机已对该应用开启定位权限
- 若后续要做持续定位或后台定位，再单独补申请；当前仅使用前台单次定位

如果你后面要走微信小程序正式发布，建议把定位权限说明写成类似：

```text
用于显示附近柜机、计算距离并帮助用户找到最近可用柜机
```

### 切到线上服务器时需要改什么

1. 把 `apps/mobile/.env.local` 或对应环境文件中的 `VITE_API_BASE_URL` 改成线上 API 地址
2. 把 `apps/admin-web/.env.local` 中的 `VITE_API_BASE_URL` 改成线上 API 地址
3. 如果后台部署域名变了，确认高德控制台里对应域名已加入允许范围
4. 重新构建后台和小程序

## 12. 图片上传保存在哪里

当前图片上传分两部分：

1. 文件本体保存到后端运行目录：

```text
runtime-uploads
```

2. 图片访问地址保存到后端数据记录中的 `imageUrl` 字段

目前项目仍是内存仓储，因此“数据库字段”已经接好了，但服务重启后数据不会永久保留；只有上传文件本身还在磁盘目录里。后续如果你把仓储换成 PostgreSQL，再把 `imageUrl` 同步写入数据库即可，不用改前端上传方式。
