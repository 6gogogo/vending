# 运行与测试说明

## 1. 后端为什么会报缺少 `@nestjs/platform-express`

NestJS 的 `NestFactory.create()` 默认需要一个 HTTP 平台驱动。当前项目使用的是标准 HTTP 服务，所以必须安装 `@nestjs/platform-express`。

现在这个依赖已经补上，`apps/api/package.json` 中已经包含：

```json
"@nestjs/platform-express": "^11.1.17"
```

## 2. 后端怎么运行

### 常用命令

- 热更新开发：`npm run dev:api`
- 单次启动，不开 watch：`npm run dev:api:once`

### 什么时候用哪个

- 平时开发优先用 `npm run dev:api`
- 如果你的本机文件监听、终端权限或杀毒软件影响了 watch，再退回 `npm run dev:api:once`

### 如果现在报 `EADDRINUSE`

这不再是缺包问题，而是 `4000` 端口已经被别的进程占用了。

当前后端默认端口来自：

```ts
const port = Number(process.env.PORT ?? 4000);
```

排查方式：

```powershell
Get-NetTCPConnection -LocalPort 4000 | Select-Object LocalAddress,LocalPort,OwningProcess,State
```

查看进程：

```powershell
Get-Process -Id <进程ID>
```

如果你不想关掉已有进程，也可以直接换端口启动：

```powershell
$env:PORT=4001
npm run dev:api:once
```

或者：

```powershell
$env:PORT=4001
npm run dev:api
```

## 3. 电脑端后台怎么运行

- 启动后台：`npm run dev:admin`
- 构建后台：`npm run build --workspace @vm/admin-web`

运行后它是普通网页后台，不是 Electron，不会出现桌面应用外壳。

## 4. 移动端现在到底是什么运行方式

当前 `npm run dev:mobile` 对应的是：

```bash
uni -p h5
```

这表示它运行的是 **H5 版本**，本质上是浏览器网页，不是“模拟一个带手机边框的真机界面”。

所以你现在看到“只有一个背景”的情况，通常意味着：

- 你打开的是 H5 页面
- 页面逻辑可能没加载成功，或者
- 你没有进入正确页面流程

它不是手机模拟器，只是把 uni-app 输出成网页来预览。

### 移动端接口地址该怎么配

移动端默认会读取：

```text
VITE_API_BASE_URL
```

示例文件已经放在：

```text
apps/mobile/.env.example
```

如果你只是用电脑里的微信开发者工具模拟器，可以写：

```env
VITE_API_BASE_URL=http://127.0.0.1:4000/api
```

如果你要用手机真机调试，就不能写 `localhost` 或 `127.0.0.1`，而要改成你电脑的局域网 IP，例如：

```env
VITE_API_BASE_URL=http://192.168.1.20:4000/api
```

原因是：

- 开发者工具模拟器运行在你电脑上
- 真机运行在手机上
- 手机里的 `localhost` 指向手机自己，不指向你的电脑

## 5. 能不能导入微信小程序测试

可以，推荐这样做。

如果你之前导入时报：

```text
在项目根目录未找到 app.json
```

原因不是微信开发者工具本身有问题，而是当时项目里缺少微信小程序平台编译包，生成出来的还是 H5 页面，不是真正的小程序工程。

现在已经补上这些依赖：

- `@dcloudio/uni-mp-weixin`
- `@dcloudio/uni-mp-alipay`
- `@dcloudio/uni-app-plus`

### 方式 A：直接开发模式输出微信小程序

在项目根目录运行：

```bash
npm run dev:mobile:weixin
```

这会让 uni-app 以微信小程序模式输出开发产物。

### 方式 B：先构建，再导入微信开发者工具

在项目根目录运行：

```bash
npm run build:mobile:weixin
```

然后用微信开发者工具导入生成目录。

## 6. 微信小程序导入步骤

1. 先安装并打开“微信开发者工具”
2. 执行 `npm run build:mobile:weixin`
3. 在微信开发者工具中点击“导入项目”
4. 项目目录选择：

```text
apps/mobile/dist/build/mp-weixin
```

5. `AppID` 如果暂时没有正式值，可以先选择测试号或无 AppID 体验
6. 导入后直接编译预览

### 真机调试额外检查项

如果开发者工具里正常，但真机白屏或请求失败，优先检查：

1. `VITE_API_BASE_URL` 是不是还写着 `localhost`
2. 手机和电脑是否在同一局域网
3. 后端是否真的在对应端口启动
4. 微信开发者工具是否开启了“不校验合法域名、web-view（业务域名）、TLS 版本以及 HTTPS 证书”

### 如果点击按钮时报 `URL is not defined`

这不是你点错了，而是之前请求层里残留了浏览器环境专用的 `URL` 构造器。

现在已经改成：

- 共享请求客户端手工拼接查询字符串
- 小程序请求层只使用 `uni.request`

如果你还看到旧报错，通常说明微信开发者工具里导入的仍是旧产物。重新执行一次：

```bash
npm run build:mobile:weixin
```

然后重新导入或重新编译 `apps/mobile/dist/build/mp-weixin`。

## 7. 支付宝小程序怎么测

- 开发模式：`npm run dev:mobile:alipay`
- 构建模式：`npm run build:mobile:alipay`

如果你之前报：

```text
no such file or directory, open '...mp-alipay\\app.json'
```

也是同一个原因：当时没有安装支付宝平台编译包，所以产物目录里没有真正的 `app.json`。

现在正确的导入目录是：

```text
apps/mobile/dist/build/mp-alipay
```

构建后再导入支付宝小程序开发工具即可。

## 8. App 怎么拿到安装包

### 先生成 App 构建产物

在项目根目录运行：

```bash
npm run build:mobile:app
```

或者：

```bash
npm run build:app --workspace @vm/mobile
```

生成目录是：

```text
apps/mobile/dist/build/app
```

### 这个目录是什么

它是 uni-app 的 App 构建产物目录，包含：

- `manifest.json`
- `app-config.js`
- `app-service.js`
- 页面样式和运行资源

它 **不是直接可安装的 `.apk` 或 `.ipa` 文件**。

### 真正的安装包怎么来

如果你要 Android 安装包，常见做法是：

1. 用 HBuilderX 打开 uni-app 项目
2. 选择“发行”
3. 选择原生 App 云打包
4. 配置 Android 包名、证书、图标、启动图
5. 让 HBuilderX 云打包生成 `.apk` 或 `.aab`

如果你只是先看 App 运行效果，可以先用 HBuilderX 导入：

```text
apps/mobile/dist/build/app
```

如果你要继续做正式发包，建议回到项目源目录 `apps/mobile` 配置完整的 `manifest.json`、证书和图标资源后再打包。

## 9. 我建议你现在怎么测

如果你主要是看页面和交互，不建议先盯着 H5。

更建议你：

1. 后端先用 `npm run dev:api:once`
2. 移动端直接用 `npm run build:mobile:weixin`
3. 导入微信开发者工具看真实小程序页面
4. 后台单独用 `npm run dev:admin` 看桌面端

## 10. 如果本地还没打通，能不能先用测试平台

可以，但建议只让后端或 `sandbox` 去连测试平台，不建议让小程序直接连。

更稳的方式有两种：

1. 把后端环境变量 `SMARTVM_BASE_URL` 指到 `http://pre.smartvm.cn`
2. 直接执行 `sandbox` 命令调测试平台，例如：

```bash
npm run sandbox:goods
npm run sandbox:door
npm run sandbox:payment-success
npm run sandbox:refund
```

不要直接让小程序去请求测试平台的主要原因是：

- `clientId` 和签名密钥不该暴露在前端
- 小程序还要额外处理请求域名白名单
- 后续正式环境也应该由业务后端统一中转柜机接口
